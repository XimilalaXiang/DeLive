# DeLive UI UX Prioritized To-Do List

> 生成日期: 2026-03-10
> 当前判断: DeLive 的下一阶段前端不应继续做“局部美化”，而应推进一次以 `桌面工作台化` 为核心的 UI/UX 重构
> 当前产品判断: DeLive 已经不只是“实时字幕工具”，而是在向 `local-first 音频智能工作台` 演进
> 本版原则: `信息架构优先`、`工作流优先`、`先重构主工作区再做细节抛光`
> 当前执行策略: 先解决主界面、会话详情、历史检索、设置中心四块结构性问题，再统一视觉系统与交互系统
> 明确不做: 本阶段不以“新增更多主题色”“补更多 hover 动画”“局部换卡片样式”为主要目标

## 这份文档的目标

这份 TODO 不是“前端小优化清单”，而是用于指导 DeLive 前端下一阶段 UI/UX 升级顺序的执行文档。

核心目标只有 4 个：

1. 把首页从“纵向堆叠页面”改造成真正的桌面工作台
2. 把会话详情从“大而杂的弹窗”改造成可持续扩展的 AI review workspace
3. 把历史记录从“基础列表”升级为可检索、可筛选、可管理的资料库入口
4. 建立统一设计系统，结束当前“局部好看但整体割裂”的状态

## 当前主要问题判断

### 1. 结构问题大于视觉问题

- 当前首页仍是 header + transcript + recording + history 的垂直堆叠
- 这更像 demo page，不像桌面生产力工具
- 真正的核心问题是信息架构和工作流编排，而不是颜色不够高级

### 2. 会话详情已经超出弹窗承载能力

- 预览弹窗里已经包含 transcript、AI briefing、Ask This Session、mind map、speaker rename、导出动作
- 这说明它已经是一个完整 workspace，而不是 modal
- 后续继续往弹窗里叠功能只会越来越乱

### 3. 设置中心负担过重

- provider 配置、本地 runtime、AI post-process、导入导出、自动更新、诊断都堆在同一个小弹窗里
- 当前体验不适合高频调整，也不适合初次理解

### 4. 视觉系统没有真正收口

- 现有主题、局部卡片、弹窗、字幕层、toast 的风格语言不完全一致
- 主应用与字幕 overlay 甚至接近两套系统
- 这会让产品显得“功能很多，但不像一个完整产品”

## 排序原则

### 优先级判断标准

按下面顺序排序：

1. 是否直接改变用户的主工作流体验
2. 是否解决结构性混乱，而不是只做表面优化
3. 是否能为后续 AI 能力继续扩展留出空间
4. 是否能统一主应用、历史、详情、设置、字幕几套体验
5. 是否会显著降低未来前端维护成本

### 当前不优先做的方向

- 新增更多主题色
- 做大量装饰性动画
- 单独重画 icon 而不处理结构
- 继续往现有弹窗里堆功能
- 在不改信息架构的前提下做大面积 CSS 美化

原因：

- 这些都无法解决当前 UI/UX 的主矛盾
- 当前最需要的是“产品层面的界面重构”，不是“视觉化妆”

## P0 必须立即推进

### 1. 重定义主信息架构与导航模型

- [ ] 明确 DeLive 的一级结构：`Live Desk / Sessions / Review / Settings`
- [ ] 不再让首页承担所有功能入口
- [ ] 确定主导航应采用顶部工作区切换、侧栏导航或混合方案
- [ ] 为后续 AI 工作台和跨会话能力预留稳定入口

为什么现在做：

- 不先重做 IA，后面所有界面优化都会继续建立在错误结构上
- 这是决定是否需要前端重构的根问题

依赖：

- 现有功能地图梳理
- 会话生命周期与 AI 工作流梳理

完成标准：

- 能清楚回答“用户现在在哪个工作区、下一步去哪里、每个区域负责什么”
- 首页不再承担设置、历史、详情、工作台四种角色

### 2. 首页重构为桌面工作台 `Live Desk`

- [ ] 将主界面从纵向堆叠改为真正的工作区布局
- [ ] 让 transcript 成为主要舞台区域，而不是页面中的一张卡片
- [ ] 将录制动作、provider/source 状态、caption controls 重组成稳定控制区
- [ ] 为 idle / recording / starting / error / recovery 定义统一状态布局

为什么现在做：

- 首页是用户停留时间最长的界面
- 当前 UI 最直接拉低产品质感的就是首页结构像 demo 而不像工具

依赖：

- `P0 #1` 的导航与工作区定义

完成标准：

- 用户打开应用后能立即感知“这是工作台，不是设置页套了几个卡片”
- transcript 区域、控制区、辅助区的层级清楚

### 3. 会话详情从弹窗升级为 `Review Desk`

- [ ] 将 session detail 从 modal 迁移为独立工作区或全尺寸详情页
- [ ] 将 AI briefing、Q&A、mind map、transcript 拆为稳定分区
- [ ] 设计 transcript-first 与 AI-first 两种浏览路径
- [ ] 为 speaker、citations、导出动作建立固定位置

为什么现在做：

- 当前预览弹窗已经承载过多功能
- 后续任何 AI 增量能力都会继续放大这个问题

依赖：

- `P0 #1` 的工作区结构
- 已有 `postProcess / ask / mindMap` 功能

完成标准：

- 会话详情不再依赖大滚动弹窗承载所有内容
- 用户能在一个稳定空间内完成阅读、提问、整理、导出

### 4. 建立统一设计系统 v2

- [ ] 统一色彩 token、surface token、border、radius、shadow、spacing、type scale
- [ ] 收口品牌主色和语义色，不再让主题系统主导产品气质
- [ ] 定义组件级规范：buttons / inputs / cards / dialogs / tags / badges / toolbars
- [ ] 建立 light/dark 成对规则，而不是局部补 dark 样式

为什么现在做：

- 当前页面能跑，但所有模块都像各自优化过一次，没有统一母体
- 不先统一 token，后续重构会继续边做边漂移

依赖：

- `P0 #1/#2/#3` 明确主要页面结构

完成标准：

- 主应用、详情页、设置页、overlay、toast 使用同一套 token 语言
- 不再频繁出现“同一层级组件视觉不一致”

### 5. 可访问性与反馈基线补齐

- [ ] 所有对话框建立 `dialog / aria-modal / aria-labelledby` 基线
- [ ] 为错误、保存、导入导出、更新状态建立 `aria-live` 或等价反馈
- [ ] 替换原生 `alert / confirm` 为产品内统一反馈组件
- [ ] 为 icon-only buttons、键盘关闭、焦点回收建立一致规范

为什么现在做：

- 这部分属于 UI/UX 的基础设施，不是锦上添花
- 当前很多问题是“能用”，但产品完成度不高

依赖：

- 统一 dialog / toast / action feedback 组件规范

完成标准：

- 所有关键交互不再依赖浏览器默认确认框
- 键盘和读屏最基础路径可用

## P1 高优先级，第二阶段推进

### 6. 历史面板升级为 `Sessions Library`

- [ ] 从按日期折叠列表升级为更强的资料库视图
- [ ] 支持 title / transcript / translated transcript / tags / provider / AI 字段 检索
- [ ] 提供更明确的筛选、排序、批量动作入口
- [ ] 为后续 multi-session 和 project memory 预留组织方式

为什么现在做：

- 当前历史面板只解决“看见记录”，没有真正解决“管理记录”
- AI 能力已经出来，历史入口必须升级才能放大价值

依赖：

- 已有 sessions / tags / postProcess 字段
- `P0 #3` Review Desk

完成标准：

- 用户能更快找到一个会话，而不是主要靠时间和标题回忆
- 历史区具备资料库感，而不是附属面板感

### 7. 设置中心重构为 `Settings Hub`

- [ ] 将 provider、AI、runtime、app、data、diagnostics 做明确分组
- [ ] 从单一窄弹窗升级为更适合复杂配置的布局
- [ ] 提供 provider onboarding / test result / failure recovery 的清晰路径
- [ ] 让本地模型引导和 bundled runtime 引导真正成为流程，而不是长表单中的附属块

为什么现在做：

- 当前设置过于拥挤，理解成本高
- provider 越多，当前设置结构越不可持续

依赖：

- `P0 #4` 统一设计系统

完成标准：

- 新用户第一次配置 provider 不再需要在一个拥挤弹窗中来回寻找
- 高级用户能高效管理多个 provider 和本地 runtime

### 8. 字幕系统统一化

- [ ] 统一主应用与 caption overlay 的 token、排版、交互反馈
- [ ] 重做字幕样式设置面板的信息分组
- [ ] 让字幕显示模式、排版、背景、锁定状态更直观
- [ ] 为 overlay 的 hover / drag / lock / close 建立一致视觉规则

为什么现在做：

- caption 是 DeLive 很有辨识度的能力
- 但当前 overlay 和主应用风格割裂，削弱了品牌一致性

依赖：

- `P0 #4` 设计系统

完成标准：

- 用户能明显感知 overlay 属于同一个产品，而不是另一个小工具
- 字幕设置不再显得像“特殊功能面板”

### 9. 统一交互组件与状态组件

- [ ] 建立统一 empty state / loading / skeleton / error state / success feedback
- [ ] 建立统一 destructive confirm / inline status / progress / retry 组件
- [ ] 统一表单校验信息样式和位置
- [ ] 统一 toast、update notification、导入导出反馈的语言与布局

为什么现在做：

- 当前很多细节不是坏，而是不一致
- 一致性是桌面工具高级感的重要来源

依赖：

- `P0 #4/#5`

完成标准：

- 常见状态不再每个模块各做一套
- 用户对“系统现在发生了什么”有稳定预期

## P2 重要，但不要打断主线

### 10. 录制期 AI 辅助体验 `Live Copilot UI`

- [ ] 为录制中摘要、topic shift、关键提醒设计低打扰侧栏或辅助区
- [ ] 明确定义录制态下的 AI 刷新频率和视觉层级
- [ ] 避免 AI 提示抢占 transcript 主舞台

为什么放在 P2：

- 产品价值很高
- 但前提是 Live Desk 本身的工作区结构要先稳定

完成标准：

- AI 辅助不干扰录制主任务
- 用户能理解哪些内容是 transcript，哪些是 AI 洞察

### 11. Speaker Intelligence 与结构化展示

- [ ] 设计 speaker summary、speaker action items、speaker stats 的卡片结构
- [ ] 明确多说话人场景下 transcript 和摘要的联动关系
- [ ] 为 speaker rename、引用、speaker drill-down 设计统一路径

为什么放在 P2：

- 这是 DeLive 和普通转录工具差异化的重要部分
- 但必须建立在 Review Desk 稳定之后

完成标准：

- 多说话人会话的价值能被清晰看见
- 用户能围绕 speaker 维度进行阅读与整理

### 12. 项目级视图与跨会话 UX

- [ ] 设计 multi-session ask 的入口与结果页
- [ ] 设计 project / topic / collection 的组织视图
- [ ] 明确跨会话摘要、检索、引用的交互层级

为什么放在 P2：

- 这是从“单会话工具”走向“知识工作台”的关键一步
- 但必须等单会话 UI 先收口

完成标准：

- 用户能对多个 session 做连续阅读和提问
- 跨会话信息不再只能靠标签和搜索词拼凑

## P3 明确保留，但后置

### 13. 品牌展示页与对外视觉升级

- [ ] 更新 README / 官网 / 产品截图的视觉一致性
- [ ] 为主品牌建立更稳定的对外展示模板
- [ ] 统一营销材料与产品内视觉语言

为什么后置：

- 当前更紧急的是产品内工作流体验
- 外部展示应建立在产品界面重构之后

### 14. 个性化主题与高级外观定制

- [ ] 再评估是否需要保留多套主题色
- [ ] 再决定是否支持高级自定义外观
- [ ] 只在主设计系统稳定后开放个性化扩展

为什么后置：

- 当前主题多样性不是核心矛盾
- 统一性优先于可选项数量

3. 会话详情升级为 `Review Desk`
4. 建立统一设计系统 v2
5. 可访问性与反馈基线补齐
6. 历史面板升级为 `Sessions Library`
7. 设置中心重构为 `Settings Hub`
8. 字幕系统统一化
9. 统一交互组件与状态组件
10. 录制期 AI 辅助体验
11. Speaker Intelligence 展示

13. 对外品牌展示升级
14. 个性化主题与高级定制

## 一句话执行原则

> 先把 DeLive 从“功能堆叠页面”重构成“结构清晰的桌面工作台”，再追求更强的视觉表现和更多 UI 花样。
